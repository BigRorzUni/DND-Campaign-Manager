\section{Introduction}

This report describes the design and implementation of a RESTful web service for managing tabletop role-playing campaigns. The system allows Dungeon Masters (DMs) to record campaigns, sessions, encounters, and gameplay events while providing analytics derived from recorded events.

The application was implemented using \textbf{FastAPI}, \textbf{SQLAlchemy}, and \textbf{SQLite}. The system follows a layered architecture separating routing logic, business logic, repositories, validation schemas, and database models.

The system supports both manual gameplay logging and automated encounter simulation, allowing encounters to be analysed using event-derived analytics.


%------------------------------------------------
\section{System Overview}

The API provides functionality for recording gameplay activity and analysing campaign data.

The core gameplay model follows a hierarchical structure:

\begin{center}
Campaign $\rightarrow$ Session $\rightarrow$ Encounter $\rightarrow$ Event
\end{center}

\begin{itemize}
\item \textbf{Campaign}: a long-running narrative managed by a Dungeon Master.
\item \textbf{Session}: a single play session within a campaign.
\item \textbf{Encounter}: a discrete gameplay interaction such as combat.
\item \textbf{Event}: an atomic action such as damage, healing, or spell usage.
\end{itemize}

This structure allows the system to record gameplay at a granular level while enabling analytics across sessions and encounters.

The API also supports encounter participants, character management, and integration with external DnD datasets.

\subsection{API Routes}

FastAPI routes define HTTP endpoints and validate incoming requests using Pydantic schemas. These routes call service-layer methods and return JSON responses.

\subsection{Service Layer}

Services contain business logic such as encounter simulation, event processing, and analytics calculations. This layer separates domain logic from request handling.

\subsection{Repositories}

Repositories encapsulate database operations and provide a clean interface between the service layer and the database.

\subsection{Database Models}

SQLAlchemy models define the relational schema used by the application.

\subsection{Schemas}

Pydantic schemas are used for request validation and response serialisation.

\subsection{Event-Based Logging}

Gameplay interactions are stored as atomic events. Each event records an action such as damage dealt, healing received, or spell usage.

This design enables flexible analytics by aggregating event data rather than storing precomputed statistics.


%------------------------------------------------
\section{Characters and Encounter Participants}

The system distinguishes between persistent campaign characters and encounter-specific participants.

\subsection{Characters}

Characters represent player characters or non-player characters belonging to a campaign. Characters store persistent attributes such as class, level, and equipment.

\subsection{Participants}

Participants represent entities within an encounter. A participant may be linked to a character or may represent a monster or custom enemy.

Separating characters from participants allows encounter-specific state such as hit points and spell slots to be tracked independently from persistent character data.


%------------------------------------------------
\section{Encounter Simulation}

The system supports automated encounter simulation. Simulated encounters generate events using the same structure as manually logged gameplay events.

The simulation process:

\begin{itemize}
\item randomises initiative order for the encounter
\item iterates through combat rounds
\item selects available actions for participants
\item generates damage, healing, or spell events
\item determines the winning side of the encounter
\end{itemize}

Events generated by simulations are stored separately from manual gameplay events so that analytics reflect real gameplay sessions.


%------------------------------------------------
\section{Analytics Features}

Analytics endpoints derive statistics from recorded gameplay events.

Supported analytics include:

\begin{itemize}
\item damage leaderboard across campaigns
\item damage taken by characters
\item healing received
\item spell slot usage
\item total time played in sessions
\end{itemize}

These analytics are calculated dynamically by aggregating event data stored in the database.


%------------------------------------------------
\section{External Dataset Integration}

The system integrates external DnD datasets for spells, equipment, monsters, and character classes.

These datasets allow the API to provide searchable endpoints for importing official game content into encounters and characters.


%------------------------------------------------
\section{Testing Strategy}

Automated tests were implemented using \texttt{pytest}. The tests use FastAPI's \texttt{TestClient} to simulate HTTP requests against the API.

Each test runs against a temporary SQLite database to ensure isolation from development data.

Tests verify the behaviour of:

\begin{itemize}
\item campaign CRUD operations
\item session management
\item encounter lifecycle
\item participant creation
\item event logging
\item analytics endpoints
\end{itemize}


%------------------------------------------------
\section{Limitations}

Several limitations exist in the current implementation.

\begin{itemize}
\item Encounter simulation uses simplified combat logic.
\item External DnD datasets are accessed via external APIs rather than local caching.
\item Authentication and multi-user support are not implemented.
\item Other resources such as ki, rage points, and conditions are not currently tracked.
\end{itemize}


%------------------------------------------------
\section{Future Work}

Potential improvements include:

\begin{itemize}
\item Tracking of non-combat events such as rests, foraging and purchasing items
\item richer combat simulation rules
\item improved analytics visualisation
\item user authentication and multi-user campaigns
\item deployment using containerisation or cloud infrastructure
\end{itemize}


%------------------------------------------------
\section{Conclusion}

This project demonstrates the design and implementation of a RESTful API for managing tabletop campaign data. The system provides structured storage for gameplay activity and enables analytics derived from event-based logging.

The layered architecture and event-based design allow the system to be extended with additional gameplay mechanics and analytics capabilities.
